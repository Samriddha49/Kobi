Software Requirements Specification (SRS): Public Poetry RAG Critic
1. Introduction
1.1 Purpose
This document defines the requirements for a Retrieval-Augmented Generation (RAG) application that provides literary critiques of English poems. The system uses a hybrid knowledge base of original and classic poetry to offer structural and thematic feedback, designed for public deployment via a web interface.
1.2 Scope
Target Language: English only.
Knowledge Base: A dual-corpus approach containing the author's original works (at least 12 poems) and a curated selection of recognizable public domain classics.
Data Format: All reference materials will be exclusively in PDF format (.pdf).
Output: A constructive, analytical literary critique generated by a Large Language Model (LLM).
Deployment: Containerized via Docker for consistent public web access.

2. Overall Description
2.1 User Characteristics
The primary users include the application's creator (for self-critique and style analysis) and the general public (for analyzing original poetry or testing the system against classic literature).
2.2 Operating Environment
Application Environment: Containerized using Docker to ensure environment consistency across local development and cloud hosting.
Interface: A web-based graphical user interface (GUI) accessible via standard web browsers.
3. System Features (Functional Requirements)
3.1 Data Ingestion & Storage 
Requirement: The system must utilize a PDF loader to extract raw text from .pdf files located in structured local directories (data/my_poems/ and data/public_poems/).
Requirement: The system must implement a preprocessing pipeline to strip common PDF artifacts like page numbers, document headers, and footers before passing the text to the chunker.
3.2 Custom Text Chunking
Requirement: The system must use a custom LangChain text splitter.
Requirement: The splitter must preserve poetic form by chunking strictly on stanzas and line breaks, explicitly avoiding standard character-count chunking.
3.3 Vector Embedding & Retrieval
Requirement: The system must convert the chunked stanzas into vector embeddings using Google's Gemini embedding models.
Requirement: The embeddings and associated metadata must be stored locally in a ChromaDB vector store.
Requirement: Upon receiving user input, the system must retrieve the most structurally and thematically similar poem chunks from both corpora to serve as contextual grounding.
3.4 Prompt Generation & Critique 
Requirement: The system must use LangChain to construct a prompt that combines the user's submitted poem, the retrieved contextual poems, and system persona instructions.
Requirement: The application must call the Gemini API to generate the final critique.
Requirement: The application must gracefully handle Gemini Free Tier rate limits by implementing retry logic, backoff strategies, or user-facing loading states.
3.5 User Interface 
Requirement: The frontend must be built entirely using Streamlit.
Requirement: The UI must feature a text input area for submitting poems and a designated display area for the AI-generated critique.
4. Non-Functional Requirements
4.1 Technology Stack
Frontend: Streamlit
Orchestration: LangChain
Data Parsing: pypdf (for PDF text extraction)
Vector Store: ChromaDB
LLM & Embeddings: Gemini API (Free Tier)
Containerization: Docker
4.2 Performance & Reliability
Latency: The system should aim to return a critique within 10–15 seconds of submission, network and API rate limits permitting.
Error Handling: The system must provide clear error messages if the API rate limit is exceeded, if the user submits non-English text, or if the PDF parser fails to read a heavily formatted file.
